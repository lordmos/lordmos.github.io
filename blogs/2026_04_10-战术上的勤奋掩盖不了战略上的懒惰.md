# 战术上的勤奋掩盖不了战略上的懒惰

## 写在 CC 源码泄露之后

### 2026年4月10日

在 GitHub Copilot 出现之后，我就一直在用这个工具，但是对 Agent 的发展后知后觉，这两年我都只是把 Copilot 当作一个代码补全工具，甚至有一天 VSCode 出现了右边栏对话框，我还在想“这不就是个聊天机器人吗？有什么了不起的”。直到我的朋友告诉我（实际上带我走了一次 Vibe Coding），“现在是这样编程的”，我才知道，我当了许久灯下黑。因此在年后，很多想法快速涌现，才有的年后 Vibe 出一堆的东西，并且对 Agent 产生了浓厚的兴趣。

所以 Claude Code 源码泄漏事件时，作为吃瓜的一份子，以及“发现 Agent 这个新大陆”的 Vibe Coding 新人，在网上搞到了一份源码——他们是顶尖的 AI 团队，我要看看他们是怎么工作，代码是怎么写的。

但是读代码不能说不是我的专长吧，至少有了 AI 之后我也学会了偷懒。“让 AI 总结一下”应该是不少“ AI 时代”人的通病，我也不例外。 [撰经阁——Scriptorium](https://lordmos.github.io/scriptorium/) 就是这次“让 AI 总结一下”的副产品。我当时是这么告诉 AI 的，下面也是`Scriptorium`的第一行 Prompt：

> 请根据 claude code 这份源代码帮我写一本《Claude Code源码编程思想》的书，使用 team agent 构建大纲，然后构建 task list 逐步完成所有章节的内容编排，所有章节中需要用多个 team agent 共同运行，使用 brainstorm ，文章内容结构参考《C++编程思想》。然后你需要扮演一个资深的技术专家，同时也是一个畅销书作者，用风趣幽默的文风和大量引人入胜的例子来让这本书不枯燥，深入浅出的带领读者体会Claude Code这份源码的巧妙之处和工程实践的得失取舍。你还需要一个编辑agent帮你审核你的每一个章节。这个编辑agent常年负责各种技术类畅销书的审核，知道如何避免带来敏感的内容，也知道纠正你前言不搭后语的地方，他来纠正你来改。我们先来计划一下。你有没有什么问题？

请你忽略上面不规范的大小写，我是复制粘贴而且也懒得改了，这不重要，重要的是经过这个“编辑部”一夜的努力（烧了不少token）完成了这本书。这个编辑部甚至给不同的角色设置了导读，让我非常欣喜。其中对于前端的导读是，让我体验 React 之美，其中第四章是关于 Ink Tui 的，这章的指引是这么说的：

```
本章你将学到：
1. 为什么在终端里用React不是疯了，而是一个经过深思熟虑的架构决策
2. Ink渲染引擎的完整管线：React组件 → Reconciler → 虚拟DOM → Yoga布局 → 终端字符
3. 96个文件、近2万行代码如何构成一个完整的终端UI框架
4. Yoga布局引擎如何在80×24的终端里实现Flexbox
5. 与传统终端UI方案（blessed/ncurses）的对比

```

我心里想说，“哇， CC 团队的人实在是太有想法了。”，为了解决他们布局的问题，他们以`React Reconciler`为基础，魔改了`Yoga`引擎。书里告诉我，`Yoga`是 Meta 为 React Native 开发的 Flexbox 布局引擎，可以在不同的设备上处理好布局。另外，他们还写了`ink.tsx`——整个渲染引擎的核心，一个1,722行的类，管理渲染循环、输入处理、屏幕更新和终端状态——的东西。看完之后心潮澎湃，原来 CC 团队这么有想法，这样就能解决 React 的代码在终端中跑 TUI 的一致性问题，如何刷新，如何处理输入，如何布局等等，而且 CC 团队的 Thariq 在 X 上强调，这个是**60 FPS**的！

现在先给大家对 Terminal 中画 UI 补充一个背景知识。终端的本质是一个基于字符的二维网格（Character Grid），它通过维护一个“光标（Cursor）”位置，并解析特殊的控制字符来决定在哪里绘制什么内容。**移动光标，塞字符**，而不需要重新全局渲染。就这么简单。

然而，我对 CC 团队的崇拜被一个视频击碎了。这个视频是 [The PrimeTime: We've lost the Tech](https://www.youtube.com/watch?v=LvW1HTSLPEk)。我来简单介绍一下，这个老哥做了一个模拟实验，核心是看 TUI 渲染是否需要强调“我们是否需要关心 60 FPS ”，他写了一个软件，随机更新 Terminal 某一部分的文本。帧数？什么帧数？我录屏快到视频都开始模糊的帧数（由于帧数高，视频中看终端非常模糊）。

老哥在视频中抨击：“你想越过高山而达到目标，不如好好想想有没有一条不需要越过高山也能达到目标的路。”

对啊，这里为什么需要 React ？为什么需要什么 Yoga ？需要一个上千行代码+引擎+渲染管线+状态管理+...的超重技术栈来维护这个终端渲染？因为你们团队的前端人员只会 React 吗？很多时候“屎山”确实是人为创造的。我们因为自己不会，所以觉得难，或者“反正 AI 能帮我解决问题”，然后理所当然创造了更多的问题。我们应该正视问题，想想这个问题是否真的要这么做，而不是“你看这很酷，我用 React 在终端做渲染” CC 团队的人出来说，“我们有 60 FPS”，就是一个笑话—— **理论上上千 FPS 都没问题的东西，你炫耀什么60 FPS？** 这就是典型的 **“想通过战术上的勤奋掩盖战略上的懒惰”** 。

这本书瑕不掩瑜，帮我快速过了一遍整个 CC 源码。我觉得可笑的是我自己，这章节的导读对这个“超重技术栈”大加赞赏，实际上只是因为我`prompt`中的那句“体会Claude Code这份源码的巧妙之处和工程实践的得失取舍”。我犯了这样一个错误：其实我在 Prompt 中已经给出了“答案”，AI 只是在帮我把答案写成书而已。我告诉他这个源码是精妙的，他就“觉得”是精妙的。因为他只有输入，没有价值判断。结果我在拿到这本书之后，我竟然还觉得“哇， CC 团队的人实在是太有想法了。”，确实是略显天真了，没有看到那个老哥的视频，我可能甚至还会去跟人到处给 CC 的技术团队做布道，恐怕就要贻笑大方了。

以此同时，我发现身边那些“即害怕自己被取代，又抱怨 AI 不好用”的人，在使用 AI 的时候也跟一个啥都不懂的老板一样，经常提出“我想要一个五彩斑斓的黑”这种令人发笑的需求。不要笑，我当时也是这么想的，我不相信我的读者里没有“帮我做一个量化工具”、“帮我做一个炒股软件”这种需求。AI 给你写了一堆“屎山”，糊弄一下你，你发现不好用，然后不了了之并且痛骂。我想他们在写下“帮我做一个量化工具”这几个字的时候恐怕甚至没听说什么叫“截面排序”。没错，AI 只是一个工具，它不能替代专家，也不能替代我们自己的思考。就跟你作为一个什么都不懂的老板，就可能就会被手下哄骗，是一个道理。AI 冲击的，就是这些“想通过战术上的勤奋掩盖战略上的懒惰”的人。

AI 能解决的其实是，你对某一个领域有足够“审美”或者知道如何具体去做一件事情，交给他去做，帮助我们摆脱陷入“细节陷阱”，而在我们不关心的一些事情上，做一个能糊弄过去的方案仅此而已。所以弄懂技术，或者弄懂业务，成为某一个领域的专家，这是我们不能偷懒的地方，才是我们的价值所在。这个叫“人的战略价值”。而即使是没有 AI 的时代，能真正跑出来的不也是这些有战略价值的人吗？

以上就是我做完`Scriptorium`的感悟，也感谢大家支持这个项目。
